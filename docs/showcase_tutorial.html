What is EIDO Sentinel?

EIDO Sentinel is an advanced Proof-of-Concept (PoC) AI agent engineered to dramatically improve how emergency communications and incident reports are managed. By harnessing the capabilities of Large Language Models (LLMs) and modern data processing, EIDO Sentinel offers a more efficient, intelligent, and responsive framework for public safety operations.

It tackles the challenge of diverse data formats, information overload, and the need for rapid sense-making in critical situations. Whether dealing with structured NENA EIDO JSON messages or unstructured raw text alerts, EIDO Sentinel transforms raw data into actionable intelligence.

Tutorial: Using the EIDO Sentinel Demo (Streamlit UI)

This guide will walk you through the EIDO Sentinel interactive demo. For non-technical users, think of this as a smart assistant that helps make sense of emergency reports quickly!

Getting Started

  1. Installation: Ensure EIDO Sentinel is installed by following the `README.md` ("Getting Started").
  2. Environment Setup (`.env`):
    • Make sure the `.env` file exists in the project root. Copy from `.env.example` if needed.
    • Configure `LLM_PROVIDER` (e.g., "google", "local") and the corresponding API keys/URLs. LLM features require a working backend.
    • Set a unique `GEOCODING_USER_AGENT` (e.g., `EidoSentinelDemo/1.0 (your.email@example.com)`).
  3. (Recommended) RAG Index: For best LLM performance, build the RAG index: `python utils/rag_indexer.py`.
  4. Start the App: Open your terminal, navigate to the `eido-sentinel` directory, activate your virtual environment (`source venv/bin/activate`), and run: 
    ./run_streamlit.sh
    Or: 
    streamlit run ui/app.py
    Access the app in your browser (usually `http://localhost:8501`).

Demo Walkthrough

1. Initial Setup & LLM Configuration

Familiarize yourself with the dashboard. In the sidebar:

  • Admin Actions: (Optional) Click "Clear All Incidents" to start fresh.
  • Configure Agent: Expand this. Select your "LLM Provider". If using Google, enter your API Key. If local, ensure your local LLM server (like Ollama) is running and the Base URL/Model Name are correct. This lets you change AI brains on the fly!
[Image: Screenshot of the 'Configure Agent' sidebar section]

2. Ingesting EIDO JSON Data

This is for structured, machine-readable reports.

  • In the sidebar, go to "Data Ingestion" -> "EIDO JSON" tab.
  • You can:
    • Upload File(s): Click to upload one or more EIDO JSON files.
    • Paste JSON: Paste EIDO JSON text directly into the area.
    • Load Sample: Choose a pre-loaded sample file (e.g., `Sample call transfer EIDO.json`).
  • Click "Process Inputs".
  • What to Expect: A success message. The dashboard numbers (Total Incidents) will update. The new incident will appear in the "List" tab and as a dot on the "Map".
[Image: Screenshot of the EIDO JSON ingestion tab with a sample selected]

3. Ingesting Raw Text Alerts

This is where the AI shines ‚Äì understanding plain text!

  • Go to "Data Ingestion" -> "Raw Text" tab.
  • Single Event: Paste a simple alert like: 
    UNIT ALERT: Engine 4 responding to report of smoke in building at 555 Oak Avenue, Springfield, ZIP 98765. Time: 14:32 PST. CAD ID: SFD24-00123.
  • Click "Process Inputs".
  • What to Expect: Success message. Metrics update. New incident in List/Map.
    • AI Parsing Power: Go to "Details" tab, select this new incident. See the AI's summary. Then, go to "EIDO Explorer", select the incident and its report. You'll see the JSON the AI *created* from your text, with extracted location, type, ZIP code, etc!
  • Multi-Event Text: Try pasting text with several alerts, like: 
    15:10: MVA I-90 EB @ MP 15. Two cars. Ref# ABC111. // 15:15: New call: Fire alarm, 123 Pine St, Redmond. E1 dispatched. Ref# XYZ222
    The AI will try to split and process these as separate events. Check the "Processing Log" in the sidebar.
  • Correlating Updates: Paste an update to an existing incident (e.g., an update to the "555 Oak Avenue" fire). 
    UPDATE: Incident SFD24-00123 at 555 Oak Avenue. Fire knocked down. Requesting ventilation. Time: 14:55 PST.
    The "Total Incidents" should *not* increase. The original incident will be updated with a new report, and its AI summary will evolve.
[Image: Screenshot of the Raw Text ingestion tab and the EIDO Explorer showing LLM-parsed JSON]

4. Exploring the Dashboard

  • Filtering: Under "Filter & Analyze Incidents", use the dropdowns to filter by Type, Status, or ZIP Code. Notice how the List, Map, and Charts update instantly. This helps you focus on what's important.
  • Map Tab: Zoom and pan. Hover over map points for quick incident info.
  • Charts Tab: See distributions of incident statuses and types.
  • Details Tab: Select different incidents. See their AI-generated summary, recommended actions, and the history of all reports linked to them.
[Image: Collage of Dashboard tabs - Map with points, List view, Charts]

5. Generating Warnings

  • Go to the "Warnings" tab.
  • The incidents currently shown (based on your filters) will be used.
  • Choose a severity, add an optional custom message.
  • Click "Generate Warning".
  • What to Expect: A formatted warning message is created, summarizing the selected incidents, ready to be copied for dissemination.
[Image: Screenshot of the Warning Generation tab]

6. Creating EIDO with the Generator Tool

  • Go to the "EIDO Generator" tab.
  • Select a template (e.g., `traffic_collision.json` ‚Äì these are examples you can create).
  • Describe a scenario in plain text, e.g., "Two-car accident at Maple & 5th, 11:45 AM EST. Minor injuries. Unit P-12 on scene. CAD: CAD2024-55001. Coords: 40.7128, -74.0060. Zip: 10007."
  • Click "Generate EIDO from Template".
  • What to Expect: The AI fills the template based on your scenario, producing a structured EIDO JSON. This is great for creating standardized test data or initial reports.
[Image: Screenshot of EIDO Generator with template and scenario]

Tutorial: Using the EIDO Sentinel API

For developers and system integrators, EIDO Sentinel offers a FastAPI-based API for programmatic interaction.

Base URL: If running locally, the API is typically at `http://localhost:8000/api/v1` (default port 8000, check `run_api.sh` or your Uvicorn startup command).

Key API Endpoints

1. Ingest EIDO JSON Report

POST /api/v1/ingest

Send a single EIDO report as a JSON dictionary in the request body.

Example Request (using cURL): 

curl -X POST "http://localhost:8000/api/v1/ingest" \
-H "Content-Type: application/json" \
-d '{
    "eidoMessageIdentifier": "api_example_001",
    "$id": "api_example_001",
    "sendingSystemIdentifier": "ExternalSystem",
    "lastUpdateTimeStamp": "2024-05-20T12:00:00Z",
    "incidentComponent": [{
        "componentIdentifier": "inc_api_001",
        "incidentTrackingIdentifier": "API-TEST-001",
        "lastUpdateTimeStamp": "2024-05-20T12:00:00Z",
        "incidentTypeCommonRegistryText": "Suspicious Package",
        "locationReference": {"$ref": "loc_api_001"}
    }],
    "locationComponent": [{
        "$id": "loc_api_001",
        "componentIdentifier": "loc_api_001",
        "locationByValue": "<?xml version=\"1.0\" encoding=\"UTF-8\"?><location><civicAddressText>789 Market St, San Francisco, CA 94103</civicAddressText><gml:Point xmlns:gml=\"http://www.opengis.net/gml\"><gml:pos>37.7839 -122.4013</gml:pos></gml:Point></location>"
    }],
    "notesComponent": [{
        "componentIdentifier": "note_api_001",
        "noteDateTimeStamp": "2024-05-20T12:01:00Z",
        "noteText": "Unattended backpack reported near main entrance."
    }]
}'
Expected Response (201 Created): 

{
    "message": "EIDO report processed successfully.",
    "status": "Success",
    "message_id": "api_example_001",
    "incident_id": "generated_incident_uuid",
    "is_new_incident": true,
    "summary": "AI-generated summary of the suspicious package...",
    "actions": ["AI-recommended action 1", "AI-recommended action 2"]
}

2. Ingest Raw Alert Text

POST /api/v1/ingest_alert

Send raw alert text. The agent will attempt to split it into events, parse each, and process them.

Example Request (using cURL): 

curl -X POST "http://localhost:8000/api/v1/ingest_alert" \
-H "Content-Type: application/json" \
-d '{
    "alert_text": "TRAFFIC ACCIDENT: Hwy 101 at Exit 5, 2 vehicles involved, minor injuries reported. Time: 10:30 AM. // UPDATE: Hwy 101 incident, one lane blocked. Expect delays."
}'
Expected Response (201 Created or 207 Multi-Status if partial success): 

{
    "message": "Alert text processing attempted.",
    "overall_status": "Success", // or "Partial Failure"
    "processed_incident_ids": ["generated_incident_uuid1"], // List of unique incident IDs affected
    "details": [
        {
            "status": "Success",
            "message_id": "llm_parsed_event1_uuid",
            "incident_id": "generated_incident_uuid1",
            "is_new_incident": true,
            "summary": "AI summary for event 1...",
            "actions": ["Action A", "Action B"],
            "source_event_index": 1,
            "source_event_total": 2,
            "original_text_snippet": "TRAFFIC ACCIDENT: Hwy 101 at Exit 5..."
        },
        {
            "status": "Success",
            "message_id": "llm_parsed_event2_uuid",
            "incident_id": "generated_incident_uuid1", // Could be same if correlated
            "is_new_incident": false,
            "summary": "Updated AI summary for event 1 incorporating event 2...",
            "actions": ["Updated Action C"],
            "source_event_index": 2,
            "source_event_total": 2,
            "original_text_snippet": "UPDATE: Hwy 101 incident, one lane blocked..."
        }
    ]
}

3. Get All Incidents

GET /api/v1/incidents

Returns a list of all incident objects in the store.

Example Request (using cURL): 
curl -X GET "http://localhost:8000/api/v1/incidents"

4. Get Specific Incident Details

GET /api/v1/incidents/{incident_id}

Returns details for a specific incident ID.

Example Request (using cURL): 
curl -X GET "http://localhost:8000/api/v1/incidents/your_incident_id_here"

Explore the API documentation (usually at `/docs` endpoint when the API is running, e.g., `http://localhost:8000/docs`) for more details on all available endpoints, request/response schemas, and to try them out interactively.

The Power of Agentic AI Workflows

Agentic AI refers to systems that can perceive their environment, make decisions, and take actions to achieve goals. For EIDO Sentinel, this means evolving from a reactive processor to a proactive partner:

  • Proactive Monitoring & Response: An AI agent could continuously monitor incoming data streams (e.g., social media feeds for keywords, sensor networks) and flag potential incidents or escalate anomalies before human intervention.
  • Autonomous Tool Use: The agent could be empowered to use external tools and APIs. For example:

    • Fetch real-time weather data for a HAZMAT incident.
    • Cross-reference details with internal databases (e.g., known offender lists, building schematics).
  • Complex Decision Support: By combining LLM reasoning with structured data and tool outputs, the agent can assist in complex, multi-step decision-making processes, suggesting optimal response plans or resource deployments based on a holistic understanding of the situation.

Fine-tuning and Local LLMs: Precision & Privacy

For mission-critical applications like public safety, ensuring the accuracy, reliability, and privacy of AI systems is paramount. EIDO Sentinel's architecture allows for strategies to address these needs.

üéØ Fine-tuning for Mission-Critical Alerting

While general-purpose LLMs are powerful, fine-tuning them on domain-specific data can significantly enhance their performance for specialized tasks:

  • Specialized Language: Public safety involves unique jargon, acronyms, and communication styles. Fine-tuning helps the LLM understand and generate text that is accurate and familiar to personnel.
  • Improved Accuracy for Specific Events: Training on data related to specific incident types (e.g., traffic accidents, structure fires, medical emergencies) can improve the LLM's ability to extract relevant information and generate appropriate summaries or alerts for those events.
  • Reduced Hallucinations & Increased Reliability: By focusing the model on a specific domain, fine-tuning can reduce the likelihood of generating incorrect or irrelevant information, which is crucial in high-stakes environments.
  • Adherence to Protocols: Models can be trained to follow specific reporting formats or include mandatory information elements as per agency SOPs.
  • Examples: Fine-tuning could involve using historical (anonymized) incident reports, agency Standard Operating Procedures, training manuals, and relevant legal or regulatory documents.
[Conceptual Image: Graph showing improved LLM accuracy after fine-tuning on domain data]

üõ°Ô∏è Local LLMs for Uncompromised Privacy & Security

Using cloud-based LLM APIs involves sending data to third-party servers, which can be a concern for sensitive public safety information. Running LLMs locally offers a robust solution:

  • Data Sovereignty: All data, including incident reports and PII, remains within the agency's secure infrastructure. No sensitive information leaves the premises.
  • Enhanced Security: Reduces the attack surface associated with transmitting data over the internet to external services.
  • Compliance: Helps meet stringent data privacy regulations (e.g., HIPAA for medical information within incident reports, CJIS for criminal justice information).
  • Reduced Latency (Potentially): Local processing can eliminate network latency associated with API calls, leading to faster responses (dependent on local hardware).
  • Cost Control (Long-Term): While requiring an initial investment in hardware, local LLMs avoid per-token or per-call API fees, potentially leading to cost savings at high volumes.
  • Offline Capability: Local models can operate even if internet connectivity is disrupted, crucial for disaster scenarios.
  • Considerations: Requires significant computational resources (powerful GPUs with ample VRAM), IT expertise for setup and maintenance, and careful selection of open-source models suitable for local deployment. EIDO Sentinel supports local LLMs through interfaces like Ollama or LM Studio.
[Conceptual Image: Server rack icon with a lock, symbolizing secure local LLM deployment]

By combining fine-tuning with the option for local LLM deployment, EIDO Sentinel can be adapted to meet the specific needs of public safety agencies, balancing cutting-edge AI capabilities with the critical requirements for accuracy, privacy, and security.

EIDO Sentinel is a foundational step. The vision extends to a comprehensive AI partner for emergency services:

  • Proactive Intelligence: Monitoring diverse data streams to anticipate and flag potential incidents before they escalate.
  • Multi-Modal Understanding: Processing not just text, but also voice (radio/911 transcripts), images (CCTV, drone footage), and sensor data for richer situational awareness.
  • Advanced Agentic Reasoning: Enabling the agent to perform complex, multi-step planning and tool utilization to manage evolving incidents with greater autonomy.
  • Continuous Learning & Adaptation: Incorporating feedback loops and adaptive learning to continuously improve parsing, summarization, and recommendation accuracy over time.
  • Seamless System Integration: Deeply embedding with CAD, RMS, alerting platforms, and other public safety systems for a truly unified operational environment.
  • Explainable AI (XAI): Providing clear, understandable justifications for its correlations, summaries, and recommendations to build trust and facilitate human oversight.

The journey is towards creating AI systems that are not just tools, but true collaborators in the critical mission of public safety.
